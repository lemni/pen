# Pen — Core Architecture Charter

**Status:** Proposed reset

---

## Goal

Define the durable architecture for Pen as a fast, headless, extension-first editor engine.

This document is intentionally narrower than the previous `v01`. It is the architecture charter for the parts that should remain stable even as higher-level product ideas change.

Pen should optimize for:

- speed
- predictable performance
- great developer experience
- reviewable, trustworthy mutation
- clean package and runtime boundaries

---

## Scope

This charter defines:

- the architectural center of Pen
- the package and runtime boundaries that matter most
- the invariants for mutation, validation, and extension composition
- the boundary between editor infrastructure and downstream product UX

This charter does not attempt to fully specify:

- chat shells
- AI workflow chrome
- planning products
- agent personas
- app-specific product analytics
- broader ecosystem packaging beyond the core seams

---

## Core Thesis

Pen is a headless editor engine with three stable layers:

1. **Contracts**: types, schemas, ops, transport contracts, and extension interfaces.
2. **Runtime**: editor authority, apply pipeline, schema normalization, selection, document state, and extension dispatch.
3. **Composition**: document tools, AI extensions, rendering primitives, transports, and other feature packages built on top of the core runtime.

The architecture should center on the smallest abstractions that are already real in the codebase:

- `DocumentOp`
- `Editor`
- `Extension`
- `DocumentState`
- `BlockHandle`
- `StructuredTargetDescriptor`
- `ToolRuntime`
- `PenTransport`
- `PenStreamPart`

If a concept cannot be explained in terms of those seams, it probably does not belong in Pen's architectural center.

---

## Non-Goals

Pen is not trying to be:

- a built-in chat product
- a mode-heavy AI shell
- a general-purpose coding agent
- a workflow orchestration layer for `/web` or `/api`
- a product analytics system
- a second document model alongside the canonical block-native one

Pen may support those experiences downstream, but they are not the architecture drivers.

## Architecture Anchors

### `DocumentOp` is the mutation currency

All durable document writes flow through `DocumentOp[]`.

This is the canonical mutation contract for:

- user actions
- AI mutations
- imported content
- programmatic edits
- collaborator changes entering the editor boundary

Any optimization, transport, or AI abstraction must eventually resolve into the same mutation currency.

### `Editor` is the authority boundary

`Editor` owns:

- `apply(ops, options)`
- the apply pipeline
- schema validation and normalization
- document profile enforcement
- extension lifecycle
- selection state
- document state
- event emission

No feature package should bypass the editor mutation pipeline for durable writes.

### `Extension` is the composition model

Behavior is added through extensions, not through one-off side channels.

Extensions may contribute:

- lifecycle hooks
- decorations
- key bindings
- input rules
- extension state
- shared runtime slots

This keeps Pen headless and composable while letting higher-level packages build rich behavior on a common host.

### `DocumentState` and `BlockHandle` are the read model

The canonical read surface for editor-aware features is:

- `DocumentState`
- block handles
- structured target inspection

That read model should stay incremental, cheap to query, and safe for tools and AI.

### `StructuredTargetDescriptor` is the AI-safe editing seam

AI should not reason about the document only as raw prose.

The structured target layer defines:

- what a target is
- whether it is editable
- what operations are valid for it
- what specialized constraints apply

This is the right bridge between the editor runtime and structured AI mutation planning.

### `ToolRuntime` and `PenTransport` are seams, not product shells

Pen exposes tool execution and model transport as infrastructure seams:

- `ToolRuntime` and `ToolContext` for document-aware tools
- `PenTransport`, `PenStreamRequest`, and `PenStreamPart` for model execution

These are deliberately smaller than an end-user agent shell.

## Package Boundaries

### `@pen/types`

`@pen/types` is the contract surface.

It should own:

- editor, selection, extension, schema, CRDT, and transport interfaces
- document operation types
- branded ids and lightweight helpers
- tool and model contracts

It should not own runtime-heavy behavior.

### `@pen/core`

`@pen/core` is the runtime authority.

It should own:

- editor creation
- apply pipeline
- schema engine integration
- document state
- selection
- decoration aggregation
- extension dispatch
- document profile policy enforcement

`@pen/core` is allowed to provide a good default experience, but the architecture should still treat it as the host rather than the owner of every feature concern.

### `@pen/document-ops`

`@pen/document-ops` is the document tooling seam.

It should own:

- document read helpers
- structured target inspection
- document write helpers that compile to `DocumentOp[]`
- tool runtime integration for editor-aware tools

It should not redefine editor authority.

### `@pen/content-ops`

`@pen/content-ops` is the content compilation seam.

It should own parse and write-content compilation logic that turns structured or text input into valid document operations.

### `@pen/ai`

`@pen/ai` is an extension-owned AI mutation layer.

It should own:

- routing and execution for AI-assisted mutation
- suggestion and review infrastructure
- receipts, step metadata, and provenance
- structured preview and plan execution bridges
- tool escalation when document-native mutation is insufficient

It should not own:

- chat product shells
- product workflows
- organization-level prompt systems
- product analytics

### `@pen/react`

`@pen/react` is the primary rendering and composition layer.

It should expose:

- headless editor primitives
- hooks that map runtime state into React
- review-oriented primitives where they are genuinely infrastructural

It should avoid becoming the architectural center of Pen.

### Adapter and transport packages

CRDT adapters, transports, importers, and exporters are important seams, but they are downstream from the core contracts and runtime.

Examples:

- `@pen/crdt-yjs`
- `@pen/transport-direct`
- `@pen/transport-sse`

## Mutation Model

### Canonical path

The canonical durable write path is:

1. construct `DocumentOp[]`
2. call `editor.apply(ops, { origin })`
3. run bounded before-apply transforms
4. validate and normalize against schema and document profile
5. transact against the CRDT document
6. update incremental read state
7. notify extensions and recompute decorations as needed

### Origins matter

`origin` is part of the contract.

Origins such as `user`, `ai`, `collaborator`, and `input-rule` are important because undo, diagnostics, trust, and policy depend on them.

### Bounded transforms only

`onBeforeApply` is useful and necessary, but must stay:

- deterministic
- bounded
- editor-local
- policy-oriented

It exists for things like:

- suggest-mode interception
- auth and policy filtering
- input-rule transformations

It is not a second mutation pipeline.

### Undo and redo grouping

Pen should selectively borrow one of the strongest foundations from `vscode-copilot-chat`: predictable undo and redo grouping around AI turns.

The architecture should support:

- one logical undo group per AI turn unless explicitly split
- coherent undo for direct apply and staged apply flows
- provenance and receipts that let developers understand what an undo group represents

Undo and redo are infrastructure concerns, not product chrome.

## AI Boundary

Pen's AI responsibility is **reviewable mutation infrastructure**.

That includes:

- document-aware routing
- direct apply vs staged review decisions
- persistent suggestions
- structured review bundles
- mutation receipts
- step metadata and execution provenance
- grouped undo and redo for AI turns
- observability for route quality, validity, and performance

That does not include:

- built-in chat shells
- visible mode systems as a core requirement
- planning products
- broad agent personas
- workflow analytics or product funnels

Pen should borrow selectively from `vscode-copilot-chat` where the foundations are strong:

- grouped undo and redo
- explicit step metadata
- execution provenance
- clear diagnostics vocabulary

Pen should not copy:

- the chat shell
- mode-heavy workflow UX
- product chrome that belongs in `/web` or `/api`

## Performance And DX

Performance and developer experience are architectural requirements, not implementation polish.

### Hot path rules

The architecture must respect that these paths are hot:

- `editor.apply()`
- schema normalization
- document state updates
- extension observation
- decoration recomputation
- AI context and structured target reads
- review entry derivation

### Design implications

The architecture should prefer:

- incremental recomputation over whole-document rebuilds
- scoped reads and writes over unbounded document passes
- explicit, serializable runtime contracts
- observability that developers can keep enabled in development
- small abstractions that compose cleanly into `/web` and `/api`

The architecture should avoid:

- duplicating heavy state in multiple layers
- product-driven abstractions in the runtime center
- unbounded hooks in the mutation path
- UI-owned semantics leaking into core contracts

## Roadmap Spine

### M0: Core editor engine

The stable architecture spine is:

1. Wave 0: contracts and foundational types
2. Wave 1: CRDT adapter layer
3. Wave 2: schema engine
4. Wave 3: editor core
5. Wave 4: transports and import/export edges
6. Wave 5: rendering and composition

These waves define Pen's core engine.

### M1: AI mutation infrastructure

The next layer is narrow and editor-centric:

- tool/runtime integration
- reviewable AI mutation
- suggestions, receipts, provenance, and diagnostics

This layer should remain subordinate to the editor architecture rather than becoming a second product architecture.

### M2 and beyond

Collaboration, layout, apps, execution, and other ecosystem ideas may still be valuable, but they should be treated as downstream feature areas.

They are not the architecture anchors.

---

## Architecture Tests

An architectural change is healthy only if it improves one or more of these without damaging the others:

- mutation correctness
- integration clarity
- performance on hot paths
- extensibility
- review trust for AI-assisted changes

If a proposal adds product surface area but weakens those fundamentals, it should not become part of Pen's architectural center.
